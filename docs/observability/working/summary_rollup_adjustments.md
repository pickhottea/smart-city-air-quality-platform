# Summary / Rollup Adjustments
Repo: `smart-city-air-quality-platform`
Status: Draft v0.1
Scope: Convert raw runtime telemetry events into curated analytical outputs

---

## 1. Purpose

This document defines the role of summary / rollup after runtime observability is introduced.

Rollup is not the source of truth.
Rollup is not a recovery mechanism for missing runtime telemetry.

Rollup exists to:

- compact runtime events
- pair related events
- generate curated CSV outputs
- improve readability for analysis and downstream review

The source of truth remains the raw runtime event layer.

---

## 2. Current-state adjustment

The current repo already has side-output telemetry generated by:

`spark/jobs/build_telemetry_metrics.py`

It currently writes outputs such as:

- `data/telemetry/pipeline_run_log.csv`
- `data/telemetry/pipeline_trace.csv`
- `data/telemetry/quality_metrics/`

After runtime observability adoption, this script must change role.

New role:

- read raw runtime telemetry events
- produce curated telemetry outputs
- perform compaction and consistency checks
- stop acting as the only way telemetry is created

---

## 3. Raw-to-curated model

### Raw runtime layer

```text
data/telemetry/runtime/run_events.jsonl
data/telemetry/runtime/span_events.jsonl
data/telemetry/runtime/metric_events.jsonl
data/telemetry/runtime/error_events.jsonl

```

### Curated analytical layer

```
data/telemetry/curated/pipeline_run_log.csv
data/telemetry/curated/pipeline_trace.csv
data/telemetry/curated/pipeline_metrics.csv
data/telemetry/curated/quality_metrics_daily.csv
data/telemetry/curated/freshness_metrics.csv
data/telemetry/curated/error_log.csv
```

## 4. Rollup principles

### 4.1 Runtime evidence first

If raw runtime events do not exist, rollup must not invent them.

### 4.2 Summarization, not recovery

Rollup may summarize, pair, compact, and normalize.

It must not pretend a missing raw event happened.

### 4.3 Deterministic output

Given the same raw event inputs, rollup should generate the same curated outputs.

### 4.4 Readability for review

Curated outputs should favor one-row-per-entity-style tables:

- one row per run
- one row per reconstructed span
- one row per metric observation
- one row per error
- one row per quality/freshness bucket

---

## 5. New role of `build_telemetry_metrics.py`

### 5.1 Allowed responsibilities

- read runtime JSONL sinks
- validate schema presence
- pair span start/end events
- compute run-level summaries
- compact metrics into curated metric tables
- derive quality and freshness summaries
- emit curated CSV outputs
- emit rollup-specific spans and metrics
- fail curated generation cleanly when needed

### 5.2 Forbidden responsibilities

- reconstruct an entire run when raw evidence is missing
- guess missing span starts or metric events
- silently overwrite missing raw telemetry with placeholders
- act as the primary telemetry producer

---

## 6. Curated output build rules

### 6.1 `pipeline_run_log.csv`

Build logic:

- group raw `run_events`
- identify one run start and one terminal state where available
- compute `started_at`, `finished_at`, `duration_ms`
- carry forward run-level error information
- optionally aggregate total rows in/out from raw metric events or final stage context

Target columns:

- `run_id`
- `trace_id`
- `pipeline_name`
- `started_at`
- `finished_at`
- `duration_ms`
- `status`
- `rows_in`
- `rows_out`
- `rows_flagged`
- `rows_rejected`
- `error_count`
- `artifact_path`
- `error_message`

### 6.2 `pipeline_trace.csv`

Build logic:

- read `span_events.jsonl`
- pair `span_start` and `span_end` by:
    - `run_id`
    - `trace_id`
    - `span_id`
- compute `started_at`, `finished_at`, `duration_ms`
- preserve parent-child links
- if start exists but end is missing, mark span as incomplete reconstruction candidate
- do not fabricate a successful span ending

Target columns:

- `trace_id`
- `span_id`
- `parent_span_id`
- `run_id`
- `pipeline_name`
- `step_name`
- `city`
- `pollutant`
- `batch_month`
- `source_name`
- `started_at`
- `finished_at`
- `duration_ms`
- `status`
- `rows_in`
- `rows_out`
- `artifact_path`
- `error_message`

### 6.3 `pipeline_metrics.csv`

Build logic:

- read `metric_events.jsonl`
- normalize field order and metric vocabulary
- write one curated row per metric event
- allow later aggregation in dashboards, but do not collapse away useful detail too early

Target columns:

- `metric_ts`
- `run_id`
- `trace_id`
- `pipeline_name`
- `step_name`
- `metric_name`
- `metric_value`
- `metric_unit`
- `city`
- `pollutant`
- `batch_month`
- `source_name`

### 6.4 `error_log.csv`

Build logic:

- read `error_events.jsonl`
- keep one curated row per raw error event
- normalize retryable and severity values

Target columns:

- `error_ts`
- `run_id`
- `trace_id`
- `span_id`
- `pipeline_name`
- `step_name`
- `city`
- `pollutant`
- `batch_month`
- `source_name`
- `error_type`
- `error_message`
- `retryable`
- `severity`

### 6.5 `quality_metrics_daily.csv`

Build logic:

- derive from Silver/Gold outputs plus runtime run context
- use actual QC results where available
- do not backfill fake metrics when QC logic did not run

Target columns:

- `metric_date`
- `run_id`
- `city`
- `pollutant`
- `total_records`
- `null_rate`
- `duplicate_rate`
- `stale_rate`
- `outlier_rate`
- `flagged_rate`
- `rejected_rate`
- `pass_rate`
- `quality_status`

### 6.6 `freshness_metrics.csv`

Build logic:

- derive from latest observed timestamps and ingestion/runtime timestamps
- compute lag in minutes
- assign freshness status from lag rule

Target columns:

- `metric_ts`
- `run_id`
- `city`
- `pollutant`
- `latest_observed_at`
- `latest_ingested_at`
- `freshness_lag_minutes`
- `freshness_status`

## 7. Span pairing rules

### 7.1 Matching key

A span start and span end are matched by:

- `run_id`
- `trace_id`
- `span_id`

### 7.2 Valid pair

A valid reconstructed span has:

- one start event
- one end event
- end time greater than or equal to start time

### 7.3 Missing end event

If a start exists but no end exists:

- keep the issue visible in rollup diagnostics
- do not convert it into a false success
- propagate run degradation if needed

### 7.4 Missing start event

If an end exists but no start exists:

- record a rollup consistency problem
- do not fabricate a start timestamp

---

## 8. Rollup diagnostics

Rollup should produce internal diagnostics such as:

- unmatched span starts count
- unmatched span ends count
- malformed metric event count
- malformed error event count
- missing run terminal state count

These diagnostics may be:

- stderr logs
- rollup metrics
- optional local report file

They do not replace the canonical curated outputs.

---

## 9. Run status rules after rollup

### 9.1 `success`

Use when:

- raw runtime evidence is complete
- curated rollup completes successfully
- business execution completed successfully

### 9.2 `failed`

Use when:

- business execution failed
- a core sink failed
- runtime evidence is missing in a way that invalidates the run

### 9.3 `partial_success`

Use when:

- business execution largely completed
- some non-core curated outputs failed
- raw runtime evidence still exists

### 9.4 `incomplete`

Use when:

- raw evidence exists
- curated output generation is partial or unfinished
- the run is not fully review-ready

---

## 10. Rollup failure policy

### 10.1 If curated output generation fails

- emit an error event where possible
- stop the current curated output build step
- preserve raw evidence untouched
- mark final run as `incomplete` or `partial_success`

### 10.2 If raw evidence is missing

- do not fabricate curated rows
- mark the run invalid or failed according to the contract
- make the missing evidence explicit

### 10.3 If rollup code crashes

- raw event files remain the evidence base
- rerun rollup after code fix
- do not rerun the business pipeline only to recreate telemetry if raw evidence already exists

---

## 11. Suggested execution order for rollup job

`build_telemetry_metrics.py` should execute in this order:

1. validate required runtime directory
2. validate raw sinks exist
3. read `run_events.jsonl`
4. read `span_events.jsonl`
5. read `metric_events.jsonl`
6. read `error_events.jsonl`
7. build `pipeline_run_log.csv`
8. build `pipeline_trace.csv`
9. build `pipeline_metrics.csv`
10. build `error_log.csv`
11. build `quality_metrics_daily.csv`
12. build `freshness_metrics.csv`
13. emit rollup completion event/metric

---

## 12. Minimum implementation target

The first acceptable rollup implementation must support:

- `pipeline_run_log.csv`
- `pipeline_trace.csv`
- `pipeline_metrics.csv`
- `error_log.csv`

`quality_metrics_daily.csv` and `freshness_metrics.csv` may follow immediately after if their derivation logic is not yet stable.

---

## 13. What rollup must never do

Do not:

- infer missing runtime spans as successful
- create metric rows that were never emitted
- suppress telemetry sink failures
- hide malformed raw data silently
- overwrite raw runtime evidence
- turn rollup into the only telemetry source

---

## 14. Final rule

Rollup improves telemetry usability.

It does not create runtime truth.

Runtime events are the evidence layer.

Curated outputs are the analysis layer.

Rollup is summarization, not recovery.
